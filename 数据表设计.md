# Flora Story 数据表设计

当前数据库：Neon Postgres。

## User

存储注册/登录用户。

- `id`: 用户主键
- `email`: 邮箱，唯一
- `passwordHash`: 密码哈希，只保存哈希不保存明文
- `displayName`: 昵称，后续可选
- `createdAt` / `updatedAt`: 创建与更新时间

## Session

存储登录态。

- `tokenHash`: Cookie token 的哈希
- `userId`: 登录用户
- `expiresAt`: 会话过期时间
- `createdAt`: 创建时间

## UploadAsset

存储用户上传/生成图片的归属关系，用来防止用户保存或生成不属于自己的图片。

- `userId`: 归属用户
- `url`: 图片公开访问 URL，唯一
- `storageProvider`: `local` / `cloudinary` / `r2`
- `storageKey`: 对象存储 key 或本地 key
- `kind`: `original` 或 `generated`
- `status`: `uploaded` / `used`
- `createdAt` / `updatedAt`: 创建与更新时间

## RateLimitEvent

存储轻量限流事件。

- `bucket`: 限流桶，例如登录邮箱、登录 IP、生成用户、上传用户
- `createdAt`: 事件时间

## FlowerRecord

存储用户每一束花。

- `userId`: 归属用户，登录后写入
- `title`: AI 生成标题
- `comment`: AI 锐评语
- `story`: 用户填写故事
- `actionType`: `received` 或 `sent`
- `recordDate`: 记录日期
- `style`: `watercolor` / `magnet` / `polaroid` / `original`
- `originalImageUrl`: 原图地址，本地开发时是 `/uploads/original/...`，线上 Cloudinary/R2 时是公开图片 URL
- `generatedImageUrl`: 生成图地址；原图风格时与原图一致
- `flowers`: JSON，最多 5 个主要花材与花语

## 上线建议

数据库已经从 SQLite 切换为 Neon Postgres。图片二进制文件不存入数据库，只在 `FlowerRecord` 中保存 URL。

没有银行卡时，线上图片存储建议先使用 Cloudinary：

- `STORAGE_PROVIDER="cloudinary"` 后，上传原图和 AI 生成图都会写入 Cloudinary。
- Cloudinary 会返回公开图片 URL，可以直接给前端展示和模型读取。
- 删除花束记录时，后端会清理未被其他记录引用的图片资源。

以后有付款方式后，也可以切换到 Cloudflare R2：

- `STORAGE_PROVIDER="r2"` 后，上传原图和 AI 生成图都会写入 R2。
- R2 bucket 推荐命名为 `flora-story`。
- 公开地址使用 `https://pub-xxxx.r2.dev` 或自定义域名。
